# Layered Rebuild Plan

## Summary

Project Understanding System V2 should be rebuilt around layered project
understanding instead of flat file summarization.

The current V1 prototype showed that repository scanning, parsing, relation
building, snapshot storage, and profile-based retrieval are feasible. The next
step is to redesign the knowledge model and pipeline so agents can understand a
project from architecture down to functions.

Primary consumer for V2: Review Agent.

Core reading model:

```text
Architecture
  -> Modules / Bounded Contexts
    -> Classes / Components
      -> Functions / Methods
        -> Evidence / Relations / Code snippets
```

## Why Rebuild

The V1 pipeline ingests a whole repository and stores mostly flat entities:
files, modules, symbols, relations, summaries, conventions, risks, and glossary
terms.

That shape works for a prototype, but it has several quality limits:

- file summaries are too coarse for code review
- function-level impact is not first-class
- architecture is inferred only indirectly from paths and relations
- context bundles do not naturally move from changed function to owning module
- downstream agents receive too much flat context and must reconstruct the
  hierarchy themselves
- LLM summaries can become noisy when every file is summarized the same way

V2 should make the hierarchy explicit and generate output by layer.

## V2 Architecture

The new system should be built as a clean repo/package. Keep useful ideas from
V1, but do not preserve the flat snapshot model as the core design.

Proposed package layout:

```text
src/project_understanding_v2/
  scan/
  parse/
  graph/
  architecture/
  summarize/
  review/
  storage/
  cli/
  models/
```

High-level flow:

```text
Repository Scan
-> Parse Symbols & Imports
-> Build Code Graph
-> Infer Architecture Map
-> Build Module Map
-> Build Class / Component Map
-> Build Function Map
-> Build Review Context
-> Write Layered Snapshot
```

## Knowledge Model

V2 should introduce a new layered snapshot model.

### RepositorySnapshot

Stores repository identity and ingest metadata.

Required fields:

- repo id
- repo path or remote URL
- branch
- commit SHA
- created timestamp
- scan stats
- quality stats

### ArchitectureMap

Represents the top-level shape of the project.

Required fields:

- detected layers
- entrypoints
- frameworks and runtime hints
- dependency direction
- major flows
- architecture confidence
- risk zones

Example layers:

- CLI / interface
- application / workflow
- domain / business logic
- infrastructure / adapters
- persistence / storage
- tests

### ModuleNode

Represents a logical module or bounded context.

Required fields:

- module id
- name
- path patterns
- responsibility
- owned files
- public surface
- dependencies
- key classes
- key functions
- risk markers

### ClassNode

Represents a class, component, service, controller, repository, or similar
medium-level construct.

Required fields:

- class id
- name
- kind
- file path
- line range
- owning module
- responsibility
- public methods
- state fields
- collaborators
- risk markers

### FunctionNode

Represents a function or method.

Required fields:

- function id
- name
- qualified path
- file path
- line range
- owning class or file
- owning module
- signature
- visibility
- parameters
- return type
- calls
- usages
- side effects
- risk markers
- evidence references

### RelationEdge

Represents graph relationships across layers.

Required relation types:

- `contains`
- `imports`
- `calls`
- `uses`
- `depends_on`
- `implements`
- `extends`

Each relation should include:

- source id
- target id
- relation type
- confidence
- evidence
- source location when available

### LayeredSummary

Replaces free-form file summaries as the main summary unit.

Summary levels:

- architecture
- module
- class
- function

Summaries should be structured, not only prose.

Minimum fields:

- responsibility
- important behavior
- dependencies
- side effects
- risk notes
- evidence
- generated by
- confidence

### ReviewContext

The primary output for Review Agent.

Required fields:

- changed files
- changed functions or symbols
- containing classes/components
- owning modules
- architecture layers
- direct callers
- direct callees
- reverse dependencies
- risk markers
- conventions
- review checklist
- evidence references

## Pipeline Design

### 1. Scan Repository

Classify files before parsing.

Categories:

- source
- test
- config
- docs
- generated
- dependency
- asset
- unknown

Dependency, build, cache, and generated files should be skipped or marked as
low-priority. Docs and config files should not be treated like source code.

### 2. Parse Repository

Extract structure from supported languages.

Initial language support:

- Python
- TypeScript

C# can be added after the V2 model stabilizes.

Parsed output should include:

- imports
- classes/components
- functions/methods
- calls
- symbol usages
- line ranges
- docstrings/comments
- parser errors

Parser errors must be recorded in the quality report. They should not be
silently swallowed.

### 3. Build Code Graph

Build deterministic relations before any LLM summarization.

Required indexes:

- file to symbols
- class to methods
- module to files
- source to outgoing relations
- target to incoming relations
- symbol name to candidates
- path to file

Review impact analysis depends on both forward and reverse traversal.

### 4. Infer Architecture

Infer architecture from:

- directory layout
- entrypoints
- framework files
- import direction
- naming conventions
- dependency graph

Architecture inference should output confidence and evidence. Low confidence is
acceptable if it is explicit.

### 5. Build Modules

Group files into logical modules.

Use path structure as the first heuristic. Improve with import density and
public surface detection.

Each module should summarize:

- responsibility
- public API
- dependencies
- important classes/functions
- risk areas

### 6. Build Class And Function Layers

Map classes/components and functions under modules.

Mark functions as high priority if they are:

- public API
- changed
- risky
- heavily called
- entrypoint-adjacent
- persistence/auth/external API related

### 7. Summarize Selectively

Do not summarize every function eagerly.

Eager summaries:

- architecture
- modules
- public classes/components
- risky functions
- changed functions

Lazy summaries:

- private helpers
- low-risk internal functions
- unrelated files

LLM summaries must be structured JSON validated by Pydantic. Invalid output
should retry once, then fall back to heuristic output and record a warning.

LLM adapters must raise typed errors such as `LLMError`. They must not return
error strings.

### 8. Build Review Context

Review context should start from changed files or changed symbols and walk up
and across the graph:

```text
changed function
-> class/component
-> module
-> architecture layer
-> callers/callees
-> reverse dependencies
-> risks
-> checklist
```

This is the main V2 value proposition.

## Public Interfaces

### Python API

Primary ingest API:

```python
snapshot = ingest_repository(
    repo_path="/path/to/repo",
    output_dir="./output",
    use_llm=True,
)
```

Primary review API:

```python
context = build_review_context(
    snapshot=snapshot,
    changed_files=["src/auth/session.py"],
    changed_symbols=None,
    max_depth=2,
)
```

### CLI

Required commands:

```bash
puv2 ingest <repo_path> --output-dir output
puv2 show-architecture <snapshot>
puv2 show-module <snapshot> <module_name>
puv2 review-context <snapshot> --changed-file path/to/file.py --json
```

Optional later commands:

```bash
puv2 search <snapshot> "authentication flow"
puv2 explain-function <snapshot> "module.Class.method"
puv2 quality-report <snapshot>
```

## Snapshot Output

V2 should keep file-based JSON storage for the first rebuild.

Minimum output structure:

```text
output/
  <project-name>/
    snapshots/
      <timestamp>/
        layered_snapshot.json
        quality_report.json
        architecture.json
        review_index.json
    latest.json
```

The complete layered snapshot should be loadable from `latest.json`.

## Quality Report

Every ingest should write a quality report.

Required metrics:

- files scanned
- files parsed
- files skipped
- unknown language count
- parser error count
- class count
- function count
- relation count by type
- module count
- architecture confidence
- summary coverage by level
- LLM success count
- LLM fallback count
- warnings

Fatal errors:

- invalid repo path
- storage write failure
- snapshot schema validation failure

Non-fatal errors:

- unsupported file type
- parser error in one file
- LLM failure for one summary
- low architecture confidence

Non-fatal errors should be recorded, not hidden.

## Review Agent Output

For each review request, output should include:

- summary of changed area
- changed files and symbols
- owning module and architecture layer
- impacted callers and users
- dependency direction notes
- risk markers
- evidence paths and line ranges
- recommended review checklist

Example checklist categories:

- behavior change
- API contract
- error handling
- side effects
- dependency impact
- persistence/data integrity
- authentication/authorization
- external API behavior
- tests to inspect or add

## Test Plan

### Unit Tests

Required unit tests:

- scanner classification for source/test/config/docs/assets
- parser extraction for functions, classes, methods, imports, calls
- graph builder relation creation
- reverse dependency lookup
- architecture layer inference
- module grouping
- class/function ownership mapping
- review context assembly
- LLM error handling
- structured LLM JSON validation and fallback
- snapshot read/write roundtrip
- quality report generation

### Fixture Repositories

Create fixture repositories:

```text
tests/fixtures/python_service/
tests/fixtures/typescript_app/
tests/fixtures/mixed_repo/
```

`python_service` should include:

- CLI entrypoint
- service layer
- repository/storage layer
- model/schema layer
- tests

`typescript_app` should include:

- components
- hooks
- API client
- state/service module
- tests

`mixed_repo` should include:

- source files
- docs
- config
- assets
- unsupported files

### Acceptance Criteria

V2 is acceptable when:

- `puv2 ingest tests/fixtures/python_service` writes a valid layered snapshot
- architecture output includes meaningful layers with evidence
- module output includes responsibility and public surface
- review context for a changed function includes function, class/file, module,
  architecture layer, callers/callees, reverse dependencies, risks, and evidence
- no domain-specific contamination appears unless it exists in the input project
- no mojibake appears in generated Markdown or JSON
- test suite covers the review-context path end to end

## Implementation Phases

### Phase 1: Foundation

- create new package structure
- define Pydantic models
- implement scanner
- implement snapshot storage
- implement quality report
- create fixture repos

### Phase 2: Parsing And Graph

- implement Python parser
- implement TypeScript parser
- build layered entity extraction
- build relation graph
- build reverse indexes

### Phase 3: Architecture And Modules

- infer architecture layers
- group modules
- map classes/functions to modules
- create architecture and module summaries

### Phase 4: Review Context

- implement review context builder
- add risk markers
- add evidence references
- generate review checklist
- expose CLI and Python API

### Phase 5: LLM Enhancement

- add structured LLM summarization
- validate LLM JSON
- add retry/fallback
- measure summary coverage and fallback rate

## Assumptions

- V2 will be a clean rebuild, not an incremental refactor of V1.
- Review Agent is the first optimized consumer.
- Python 3.12 and Pydantic v2 remain the base stack.
- File-based JSON snapshots remain the first storage target.
- Python and TypeScript are the initial supported languages.
- C# support comes after the core layered model is stable.
- Deterministic graph and heuristics come before LLM summarization.
- Function summaries are lazy by default unless the function is public, changed,
  risky, or important to impact analysis.
